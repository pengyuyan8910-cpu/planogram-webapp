# 全品类可视化陈列系统

一套可直接部署到 GitHub Pages 的静态网页版陈列工具。项目已经内置最后确认版全品类陈列数据，打开网页即可使用。

## 已实现功能

- 五个一级类目独立切换：厨房食品、儿童食品、酒水乳饮、日化洗护、休闲零食。
- 按二级类目和货架组展示 A/B/C/D 层陈列。
- 每个正面单独显示为一个坑位。
- 拖动任一坑位时，移动该 SKU 的全部连续坑位，避免同一 SKU 被拆散。
- 拖入新层后，货架已用长度和余量自动变化。
- 将坑位拖到“减少1个坑位”区域，可人工减少一个陈列面。
- 产品池可修改计划坑位数，例如由3个坑位改为2个坑位。
- 产品池字段修改后，陈列图上的名称、等级、尺寸、箱数、周转天数同步更新。
- 未放入SKU可加入货架，也可以替换已选中的货架SKU。
- “下掉SKU”会移除全部坑位，并自动进入未放入SKU池。
- “淘汰SKU”会移除全部坑位并进入淘汰池，数据不会删除。
- 淘汰SKU可“捞回”，捞回后进入未放入SKU池。
- 总产品池＝货架SKU＋未放入SKU，不包含淘汰SKU。
- 支持新增SKU。
- 支持当前一级类目陈列图导出为 A3 横向 PDF，每个货架组一页。
- 支持导出/导入 JSON 数据备份。
- 支持成员导出“本次修改”变更包，由管理员导入并核对冲突。
- 管理员可使用 GitHub 细粒度令牌将确认数据同步回仓库底表。
- 管理员可导出最新 Excel 底表（SKU底表、货架层、陈列坑位）。
- “恢复底表”支持密码保护，避免协作人员误初始化。
- 浏览器自动保存人工调整。

## 本地预览

### 方法一：直接打开

双击 `index.html`。大多数功能可以直接使用。PDF功能需要联网加载 `html2canvas` 和 `jsPDF`。

### 方法二：启动本地静态服务器

在项目目录执行：

```bash
python -m http.server 8080
```

浏览器打开：

```text
http://localhost:8080
```

## GitHub Pages部署

1. 新建 GitHub 仓库。
2. 将本项目所有文件上传到仓库根目录。
3. 推送到 `main` 分支。
4. 项目内置的 GitHub Actions 会自动部署 Pages。
5. 打开仓库的 `Settings → Pages`，确认 Source 为 `GitHub Actions`。
6. 等待 Actions 完成后，访问 GitHub 提供的网址。

## 数据保存说明

当前版本是纯静态网页：

- 人工调整保存在浏览器 `localStorage`。
- 同一台电脑、同一浏览器再次打开时会自动恢复。
- 不同电脑或不同浏览器之间不会自动同步。
- 建议定期点击“备份数据JSON”。
- 在另一台电脑上使用“导入数据JSON”可恢复调整结果。

## 多人协作：GitHub 审核同步模式

本项目不开放普通协作人员直接写入 GitHub 底表。推荐流程为：

1. 协作人员在小程序调整后，点击“导出本次修改”，将变更包交给管理员。
2. 仓库拥有者打开“底表管理员”，导入变更包；系统会统计 SKU、货架组改动及潜在冲突。
3. 管理员确认后，在同一页面输入仅授予本仓库 Contents: write 权限的 GitHub 细粒度令牌，点击“同步至 GitHub 底表”。令牌仅本次使用，不会写入浏览器或仓库。
4. GitHub Pages 部署完成后，其他人刷新网页即可看到统一底表；管理员可在该页面导出最新 Excel。

“恢复底表”密码由管理员在“底表管理员”中设置，并在下一次 GitHub 同步后对所有协作人员生效。该密码用于防止误操作，不应视为防御恶意篡改的安全措施。

如需多人实时同时编辑、细粒度账号权限或完整修改审计，应改用 Supabase、Firebase 或自建 API。

## PDF说明

- 导出当前一级类目。
- A3横向。
- 每个货架组单独一页。
- 基础坑位白色，扩陈坑位浅蓝灰。
- PDF组件通过 jsDelivr CDN 加载，因此导出时需要网络连接。

## 陈列规则保护

- B/C/D层的D级商品不允许产生超过基础坑位数的扩陈坑位。
- 修改计划坑位数时会检查目标层容量。
- 移动SKU时按照整个SKU模块移动，保证其坑位连续。
- 单独减少坑位使用底部“减少1个坑位”区域。
